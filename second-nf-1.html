Second Normal Form

Note on Second and Third Normal Forms

Where First Normal Form is concerned mainly about the integrity of the table format (that is: that entries are filled in; that columns are about the same attribute), Second and Third Normal Forms start to look at signs that a table has combined two concepts in a way that would be healthier if they were separated into different entities.

Both Second and Third Normal Forms ask the designer to examine attributes and confirm they describe whatever is identified by the full key: that is, that the attributes help describe the row and not inadvertently something else.

Formal Terms

The formal definition of 2NF is widely-quoted, and its terms have become part of the way people talk about database design.

In particular, the concept of “functionally dependent” is important and worth remembering by name.

The other parts of the definition play a less important role.

All are explained in this section.

Formal Definition of 2NF

“A 1NF table is in 2NF if and only if none of its non-prime attributes are functionally dependent on a part (proper subset) of a candidate key.” –E.F.Codd

Functionally Dependent

Functions and tables

A function is something that (1) takes values as input, (2) produces results, and (3) always produces the same result if given the same inputs.

The “something” might be a program, algorithm, math equation, or even an oracle or black box. Notably, it can be a lookup table, which is basically what the tables in an RDBMS are.

A function written in code:

func shouldWeEat(amIHungry: Bool, areYouHungry: Bool) -> Bool {
   if amIHungry && areYouHungry {
       return true
   }
   return false
}

The same function expressed as a lookup table:

I’m hungry I’m not hungry
you’re hungry Let’s eat! not now
you’re not hungry not now not now

Which can also be written in database table style:

My feeling Your feeling Plan
hungry not hungry eat later
hungry hungry eat now
not hungry not hungry eat later
not hungry hungry eat later

Dependency

I’m hungry I’m not hungry
you’re hungry Let’s eat! not now
you’re not hungry not now not now

In this function, the result can depend on both of the inputs. If I’m hungry, whether we eat depends on if you are hungry. If you’re hungry, whether we eat depends on if I’m hungry. The function depends on both our feelings. It is “functional dependent” on the inputs “my hunger” and “your hunger”.

Non-Dependency

A selfish person’s decision-making might work differently:

func shouldWeEat(amIHungry: Bool, areYouHungry: Bool) -> Bool {
    return amIHungry
}

As a lookup table:

I’m hungry I’m not hungry
you’re hungry Let’s eat! not now
you’re not hungry Let’s eat! not now

This is still a function: it produces a repeatable answer when given repeated inputs. It is even a function of two inputs: it does ask you how you’re feeling.

But: it doesn’t care how you’re feeling. Whatever you say doesn’t ever change the function’s output.

This selfish table is not functionally dependent on “your hunger”. All you need to know to fill in the table is the value of “I’m hungry.”

Candidate Key

A “candidate key” is any attribute or combination of attributes that could function as a key for the table.

A person table of people might use a surrogate key like a UUID to uniquely identify each person, but they might also be uniquely identified by (firstname, lastname, and date of birth).

Remember that candidate keys might be composite.

The 2NF definition uses “candidate key” to make sure you consider the different ways in which the “what is this row about” question can be answered. If you are using a surrogate key, it reminds you to to look for a natural key.

Part/Proper Subset

A “part of” is formed by dropping something, leaving a partial bit remaining.

The “proper subset” means the “something” you drop isn’t “nothing” and that it isn’t “everything.”

In the context of keys: the “part of a key” has to be smaller than the original key and still has to be able to carry meaning.

For part of the key to be dropped, the key must be composite. The “something” that is removed is one (or more) of the attributes that contribute to a (candidate) composite key.

Non-Prime Attribute

An attribute that isn’t part of a candidate key.

A non-prime attribute may help describe an entity, but is not part of its identity.

Putting it together

“A 1NF table is in 2NF if and only if none of its non-prime attributes are functionally dependent on a part (proper subset) of a candidate key.” –E.F.Codd

No attribute that is supposed to describe an element (“non-prime attribute”) should turn out to describe (be “functionally dependent on”) an aspect that is only a part of the entity’s identity (“subset of a candidate key”).

Second Normal Form

Every attribute in a table must help describe the particularly identified entity. Specifically: if the entity is identified by a composite key, then all parts of that key have to be needed to know what value should be given to an attribute. (If knowing only part of the key tells you what value an attribute should have, then the value is about that part of the key, not about the entity itself.)

2NF is only about entities that have composite keys. The composite key doesn’t have to be the primary key; it matters if there is a composite that might work as a primary key. The sub-key is a way of grouping elements. 2NF looks for attributes that are common to the groups. If there aren’t groups, there isn’t a risk that an attribute is actually about the group.

Example: students

Consider the “enrollment” entity linking students and courses, to which I propose attaching GPA, grade, and meeting time:

The table of enrollments and grades might look like this:

student course grade meets GPA
lora IT-120 4.0 TWR 8a-11a 3.9
lora IT-200 3.8 TR 2p-3p 3.9
lora IT-160 4.0 F 8a-11a 3.9
pero IT-120 3.8 TWR 8a-11a 3.5
pero IT-160 3.2 F 8a-11a 3.5
kit IT-200 3.0 TR 2p-3pp 3.2
kit IT-160 2.8 F 8a-11a 3.2
sudha IT-120 3.7 TWR 8a-11a 3.8
sudha IT-200 3.9 TR 2p-3p 3.8
sudha IT-300 4.0 F 1p-4p 3.8

Reminder: the primary key for enrollment is (student, course).

Because (student, course) is a composite key (thus: unique), we can organize a view using them, and look at each of the non-key attributes to see how they fit with 2NF.

In the tables that follow, it is OK to ignore the holes. In a many-to-many relation, nothing says you must have everything connected.to everything else.

GPA

GPA lora pero kit sudha
IT-120 3.9 3.5 3.8
IT-160 3.9 3.5 3.2
IT-200 3.9 3.2 3.8
IT-300 3.8

Notice how the GPAs in any column are all the same. This is not an accident: each student has only one GPA at a time. GPA describes the student and varying the course doesn’t matter.

Putting GPA in the enrollment table violates 2NF: it is dependent on only the student part of the composite (student+course) key.

Course Grade

grade lora pero kit sudha
IT-120 4.0 3.8 3.7
IT-160 4.0 3.2 2.8
IT-200 3.8 3.0 3.9
IT-300 4.0

Grades vary by columns and by rows. You need to know both which student and which course to look up what grade someone got.

Having grade in this table is compatible with 2NF.

Meeting time

meets lora pero kit sudha
IT-120 TWR 8a-11a TWR 8a-11a TWR 8a-11a
IT-160 F 8a-11a F 8a-11a F 8a-11a
IT-200 TR 2p-3p TR 2p-3p TR 2p-3p
IT-300 F 1p-4p

In the “meets” case, if you know only the course, you know the meeting time. It’s the same no matter what students are attending. Again, this isn’t just coincidence: the class is set up to have a common meeting time.

Putting the course meeting time into the enrollments table violates 2NF: the meeting time is not customized for each student.

In Practice

Second Normal Form seeks to clarify what the table is about (the whole key), and then to assure that the non-key attributes are being used to describe the entity and not just one of its components.

When a violation is discovered, the sub-part of the key that the attribute is actually dependent can be used to identify a different (if there’s already a table for it) or new entity. And the dependent attribute can be moved into that entity.

Fixing 2NF and 3NF Violations

With both 2NF and 3NF, the violation identifies that the entity is trying to do too much, and that it has another entity inside it.

To fix: you need to extract the inside entity and make it its own entity. (Sometimes the design already has that entity: if you find two entities have the same keys, they are about the same subject and might be combined!)

When you extract/factor/move/lift the embedded entity, you still need to remember it is associated with the original. You use the relational model to connect the two entities. This was said really well:

“It follows that this information should be moved into a separate table (Studio), and only a foreign key retained.”

–Samuel O’Neal; Winter 2023
