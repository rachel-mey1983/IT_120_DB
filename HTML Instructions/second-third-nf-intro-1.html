Second and Third Normal Forms

First Normal Form is concerned with making sure each row in a table is made up of simple values: that none of the row elements contain an embedded table. One-To-Many relationships–containers and plurals–are the typical design problem that might lead to a 1NF violation.

The Second and Third Normal Forms are concerned with detecting if copies of entities are being hidden inside other entities, where instead they should be related using a To-One relationship.

Problems!

There are duplications:

  • 26949 Database rows have lots of commonality
  • Ed’s classes duplicate email

Two extra entites have been embedded in Enrollment:

  • Class
  • Instructor

(Even though there is no problems fitting them in the table format!)

Dependencies

  • Grade -> Student and Class
  • Title -> Class
  • Instructor -> Class
  • Email -> Instructor
  • Time -> Class

Analysis

Full key is (CTCLink, Class), so only Grade should be in this table!

Class is a determinant, but Class is not the full key! It should have its own table and FKs should reference it.

Instructor is a non-key determinant: it should have its own table and FKs should reference it.

Summary

Second and Third Normal Form:

  • are elements inside other elements
  • are detected by analyzing data dependence
  • are fixed by changing to Many-to-One
