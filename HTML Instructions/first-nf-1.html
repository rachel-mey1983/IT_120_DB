First Normal Form

Formal Definition

“A relation is in first normal form if and only if no attribute domain has relations as elements.” – E.F.Codd

Translation into more familiar terms

To be in 1NF, a table must not have any values that themselves contain tables.

Plural Values

The main reason designs violate 1NF is that they are struggling to model a collection of items within an entity. You may see these thought of as:

  • a collection
  • a list (often with values delimited by commas or slashes)
  • an array
  • a plural attribute

Fixing a 1NF violation

  • move the container attribute/column(s) to a new entity/table
  • in the new table, add a FK column to relate back to the original
  • use that FK to connect the Many side (the plurals that have moved) to the One side (the original row)

Find the violation

Extra Columns

Some designs try to get around fixing a 1NF problem by adding columns to give a little more space for the data, adding attributes like “Phone 1,” “Phone 2,” “Phone 3,” etc.

These are not technically 1NF violations! However: as soon as you run out of the pre-defined extra attributes, you run into a 1NF problem.

If you see columns with numbers (or other things that might be values and not attributes, like ‘Home Phone’, ‘Work Phone’), ask if there is a chance you will run out of these pre-defined columns.

If there is any chance you might want more columns than the design shows, treat this like a 1NF violation.

Popular Definition

“There should be no multi-valued columns and no repeating groups. In addition the data in any column should all be of the same type.” –common approximation

Concerns with this definition:

  • “repeating groups” is surprisingly complicated and leaks into 2NF and 3NF
  • isn’t clear about plurals
  • isn’t clear about added columns

Practical Guidelines

  • instances of entities must fit into a two-dimensional table, with each instance as a row, and with columns as the entity’s attributes
  • there is no built-in order to instances in rows, or importance to order
  • cell values must be simple: one value per cell. Tables or lists in one cell are not allowed.
    • “overflow” attributes violate the spirit of the “simple cell”
  • the data in any column should all be the same type
    • all the instances in the table should be examples of the same entity kind
  • technically: no values can be blank (but we will be lenient about this rule)
  • fix 1NF violations by moving columns to a new table and using a 1:many relation

1NF is about giving tables structure. Values in tables should be simple and consistent.
